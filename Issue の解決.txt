＊＊リファクタリング＊＊

「リファクタリング」とは、Webアプリケーションの開発後、
ユーザーからのフィードバックをもとに新機能を追加したり、既存のコードの改善をしたりすることをいう。


---
＊＊ Slack と GitHub の連携の仕組み ＊＊

### 1. 全体像：誰が誰に「許可」を出したのか？
今回の登場人物は「Slack」「GitHub」「GitHub App（連携プログラム）」の3つ。

1.  **Slackの部屋（チャンネル）にGitHubアプリを呼ぶ**
    * `/invite @github` をしたのは、「この部屋にGitHubからの通知を持ってくる代理人を立たせる」という行為。
2.  **GitHubアカウントとの紐付け（Connect / Authorize）**
    * Slack上の自分と、GitHub上の自分のアカウントを一致させる作業のこと。
    「Authorize（認可）」することで、Slackが代わりにGitHubの情報を見に行く許可を与えた。
3.  **リポジトリの購読（Subscribe）**
    * 「どこのリポジトリの情報を、このチャンネルに流すか」という最終的なターゲット指定のこと。

---

### 2. 「Install GitHub App」とは何だったのか？
これは**「自分のGitHubリポジトリ（家）に、Slackというお客さんを入れるためのドアを設置する」**作業のこと。

* **なぜ必要か：** GitHubは非常にセキュリティが厳しいため、外部ツール（Slack）が勝手に中身を覗くことはできない。
* **何をしたか：** 「GitHub App」という橋渡し役を、自分のリポジトリ側にインストールすることで、
                    「このリポジトリの更新情報は、Slackに送信してもいいよ」という設定を有効にした。

> **豆知識：**`/github subscribe ...` と打った際、裏側で「このリポジトリにはまだドア（App）がないぞ！」と判定されると、インストールを促すURLが発行される。

---

### 3. あの「認証コード」は何のため？
「Success!」の前に打ったコードは、**「偽物がなりすまして連携していないか」**を確かめるための本人確認。

1.  **Slack側：** 「連携したい！」とリクエストを出す。
2.  **GitHub側：** 「OK、じゃあこのコードをSlackに入力して。それができれば、ブラウザを開いているお前と、Slackを使っているお前が同一人物だと認めるよ」と返答。
3.  **完了：** コードが一致することで、**「Slack ↔ GitHub」の安全な専用回線**がつながった。

---

### 4. まとめ：やったこと
時系列で整理するとこうなる。

| ステップ | アクション | エンジニア的な意味 |
| :--- | :--- | :--- |
| **① 招待** | `/invite @github` | チャンネルに「通知の受信機」を置いた |
| **② 認証** | `Connect GitHub account` / 認証コード | SlackとGitHubの「個人アカウント」を紐付けた |
| **③ 権限付与** | `Authorize Slack` | SlackがGitHubのデータを読み取る許可を出した |
| **④ 購読** | `/github subscribe GitHub ryeonthesulfur/training_curriculum_rails7` | 「このリポジトリ」の通知を流すよう命じた |

---

### Webエンジニアとしての視点
この「認可（Authorization）」の仕組みは、**OAuth（オーオース）**と呼ばれる世界標準の技術。
これから作る「精神世界アプリ」でも、例えば「Googleアカウントでログイン」や「SNS連携」を実装したくなったら、全く同じ仕組み（アプリ間の情報の受け渡しと許可）を使うことになる。

今回の「の」が入ってエラーになった経験も、「コマンドは一文字一句が正確な命令である必要がある」というAPI連携の基本を学ぶ良い機会であった。
